项目背景

现状问题

产品迭代早期，真实用户反馈往往散落在社交媒体笔记和评论区里。如果靠人工检索、复制、筛选和总结， 效率低，也容易只看到热门观点，漏掉长尾痛点。

解决方案

VoxRadar 把这条链路做成一个可复用的工作台：用户在前端输入想调研的产品或关键词，系统自动启动 小红书采集任务，抓取相关笔记与评论，整理成结构化数据，再交给分析模块生成 Markdown 报告。 前端同时展示报告和原始数据，方便一边看结论，一边回到证据。

核心特性

🔎

一键采集

输入产品名或调研问题后，系统自动启动小红书搜索采集任务，减少手动找帖和复制评论的成本。

🧠

Query 扩展

围绕原始关键词扩展更像真实用户会搜索的 query，覆盖体验、吐槽、对比、决策等反馈场景。

📊

反馈分析报告

把评论数据整理为用户反馈分析报告，突出体验较好场景、体验不好场景、需求卡和高频词。

📡

实时进度

前端支持实时展示任务进度和运行状态，让用户知道当前处于数据获取、整理还是生成报告阶段。

🔗

证据可溯

报告旁边保留原始帖子和评论证据，用户可以回看来源，而不是只看到一段不可复核的总结。

技术架构

技术栈：原生 HTML/CSS/JavaScript、FastAPI、WebSocket、Playwright、Pydantic、httpx、uvicorn、JSONL、SQLite/MySQL/MongoDB/Excel 存储适配、DeepSeek 兼容 Chat Completions API。

架构亮点：

  • 前端是一个单页分析工作台，负责收集调研输入、展示报告、展示原始证据和实时状态；后端负责采集任务编排、模型调用和数据文件管理。
  • 后端通过 FastAPI 封装爬虫启动、停止、状态、日志、数据下载和分析接口，把命令行爬虫包装成可交互的产品流程。
  • 任务运行日志和状态通过 WebSocket 推送，前端失败时还会回退到轮询，保证 Demo 场景下不会因连接中断完全失去进度反馈。
  • 分析模块支持两种路径：有模型密钥时调用 LLM 生成完整报告；没有密钥或调用失败时，使用离线规则生成基础版报告，降低演示失败风险。
  • 数据以 JSONL 为主进行沉淀，并保留 CSV、Excel、SQLite、MySQL、MongoDB 等存储适配，说明底层采集框架具备继续扩展的空间。
  • 配置项被收拢到本地 settings 文件，前端可保存 API Base、API Key、模型参数和自定义 Prompt，便于调试不同模型和报告风格。

项目进展

当前 Demo 已跑通完整产品闭环：输入关键词 → 启动小红书采集 → 保存笔记与评论数据 → 生成用户反馈分析报告 → 在前端展示报告与原始证据。

已实现的能力：前端首页可直接作为调研入口；后端提供采集任务启动、停止、状态查询、日志查询、WebSocket 推送和报告下载接口；分析模块可从 JSONL 数据生成 Markdown 报告，并支持缺少 API Key 时的离线兜底报告；前端可自动探测本地后端端口，WebSocket 失败时用轮询继续获取状态和日志。

Roadmap
